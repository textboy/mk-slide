# present-workflow

> **present-workflow** Fork 并增强自 [ppt-agent-workflow-san/ppt-workflow](https://github.com/mucsbr/ppt-agent-workflow-san/tree/main/ppt-workflow)，用于准备演示文稿的故事线作为第一阶段。

## 概述

将 PPT 创建视为分阶段的工作，而非一次性生成。本技能指导 AI agent **什么阶段该做什么**，同时将 **如何执行** 交给 agent 及其环境决定。

不规定具体的搜索工具、浏览器、MCP、渲染器或导出路径，除非特定环境明确要求。

## 核心原则

约束工作流，而非实现方式。

本技能帮助 agent 判断：
- 需求何时仍不明确
- 何时需要新的事实调研
- 何时该停下来让用户评审中间结果
- 何时继续推进到更完整的草稿
- 当能力受限时如何如实回应

## 支持的输入方式

### 1. 仅主题
示例："帮我做一个 AI 安全体检中心的 PPT"
如信息不足，询问最少缺失字段：受众、目的、页数范围、风格偏好、是否需要调研、是否需要中间评审。

### 2. 主题 + 需求说明
当用户已提供受众、目的、风格、页数和关键信息时使用。

### 3. 资料驱动的请求
当用户提供 URL、报告、PDF、笔记、转录稿或其他原始材料时使用。将用户提供的材料作为主要上下文，仅在能显著改善结果时才添加外部调研。

### 4. 现有大纲或部分草稿
当用户已有结构、早期 deck 或页面想法，希望改进而非完全重做时使用。

## 工作流

### 1. 澄清需求
收集必要的输入信息以避免方向偏差：主题、受众、目的、页数范围、风格、必含章节、应避免的内容、是否希望快速输出或分阶段确认。

### 2. 判断是否需要调研
如果主题依赖当前事件、市场状况、产品细节、统计数据或技术事实，则先调研再列大纲。如果用户明确要求研究支持的内容，或用户提供的材料明显不完整，也需要调研。否则直接从已有上下文推进。

### 3. 收集或整理上下文
使用 agent 的可用能力收集或整理支撑材料。如果无法进行外部调研，如实说明结果是资料受限的，依靠用户提供的材料而非编造确定性。

### 4. 需要时产出调研简报
在构建 deck 之前，将工作知识压缩为简报：关键事实、支撑证据、受众相关上下文、风险与未解决问题。

### 5. 先列大纲，再设计
在简报和上下文足够完善后构建 deck 结构。保持逻辑结构，确保论点与可用证据一致，使 deck 适合讲解而非仅阅读，保持大纲易于用户评审。

### 6. 添加策划稿
对非平凡 deck，不要从大纲直接跳到成品页面。创建页面级策划卡片，明确：页面要点、受众应记住什么、所需支撑信息、信息层级、推荐的视觉或结构处理方式。

### 7. 生成可评审的中间产物
在扩展之前，产出用户能给出反馈的东西，形式不限：调研简报、大纲、策划稿、示例页面或其他环境支持的可评审输出。

### 8. 重要 deck 暂停评审
对外 deck、管理层/董事会 deck、客户/sales deck、基于不确定或快速变化事实的 deck、复杂技术主题，使用显式评审关卡。

### 9. 扩展或定稿
仅在方向确认后，才向更完整的 deck 计划或更丰富的产物推进。

### 10. 审查并披露局限性
交付前检查：逻辑、事实可信度、证据覆盖、信息密度、重点与层次、跨章节一致性。如果能力限制影响了结果，如实说明。

### 11. 保存输出到文件
将最终确认的 deck 计划（或所交付的输出层）保存到 `./output/<name>.md`。如果 `./output/` 目录不存在则创建。该 markdown 文件作为流水线中下一个技能（如 `present-html`）的输入。

## 协调规则

- **调研规则**：当任务依赖事实时，agent 应在自信陈述之前收集或验证支撑材料。
- **诚实规则**：如果环境无法支持理想工作流，agent 应说明限制并在可达到的最高价值层继续。
- **评审关卡规则**：对复杂或高风险任务，agent 应在提交完整草稿前展示中间成果。
- **非假设规则**：从不假设特定的研究、渲染、预览或导出实现路径。
