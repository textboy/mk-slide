Why this mattered

Meridian Health's patient portal serves 4.2M users. Our API is the backbone.

  • ● Jan 14: PagerDuty fires at 2:17am — p99 latency spikes to 340ms on /api/v2/records
  • ● Mobile app timeout threshold: 250ms. Result: 23% of requests failing for night-shift nurses
  • ● Root cause wasn't the code — the Node.js handlers were fine. It was infrastructure friction: cold connections, no caching layer, TLS renegotiation on every request
  • ● Rewrite estimates: 4-6 months, 3 engineers. We had neither the time nor the headcount

What we considered

Option A: Full Rewrite

  • ✗ 4-6 month timeline
  • ✗ High regression risk (HIPAA audit)
  • ✗ Requires revalidation with all 3 EHR integrations
  • ✗ Team of 3 fully dedicated

Option B: Infra-Only

  • ✓ 3-week rollout (1 engineer)
  • ✓ Zero application code changes
  • ✓ Transparent to all consumers
  • ✓ Reversible — feature-flagged per route

We picked B. Not because it was fancy — because it was deployable on Monday.

The edge cache layer

src/workers/edge-cache.ts TypeScript 
export default {
  async fetch(req: Request, env: Env): Promise<Response> {
    const cacheKey = buildCacheKey(req);
    const cached = await env.KV.get(cacheKey, 'json');

    if (cached && !isStale(cached.ts, TTL_MAP[req.path])) {
      return jsonResponse(cached.data, { 'X-Cache': 'HIT' });
    }

    const res = await fetch(env.ORIGIN + req.path, {
      headers: stripHopHeaders(req.headers),
    });
    await env.KV.put(cacheKey, JSON.stringify({ data: await res.json(), ts: Date.now() }));
    return res;
  }
};

Key insight: Per-route TTLs via TTL_MAP. Patient vitals = 30s. Formulary data = 24h. Personalized, not blanket caching.

The connection pool trick

infra/pgbouncer.ini → pool-wrapper.ts Before / After 
// BEFORE: each request opens a new TCP + TLS connection
const client = new Client({ connectionString: process.env.DB_URL });
await client.connect();  // ~45ms cold start per query

// AFTER: warm pool via pgBouncer + transaction mode
import { Pool } from 'pg';
const pool = new Pool({
  connectionString: process.env.PGBOUNCER_URL,
  max: 20,
  idleTimeoutMillis: 30_000,
  connectionTimeoutMillis: 3_000,
});
// Avg connect time: 45ms → 0.8ms (pre-warmed)

The trick: pgBouncer in transaction mode keeps 20 warm connections. The Node app connects to localhost:6432 instead of RDS directly. One env var change per service.

What I'd tell past-me

  • $ Profile before you refactor. Our bottleneck was TLS handshake + connection setup, not handler logic. Flame graphs don't lie.
  • $ Per-route TTLs > global TTLs. Vitals need 30s freshness. Drug formularies can cache for 24h. One-size-fits-all caching wastes either freshness or performance.
  • $ pgBouncer in transaction mode is free performance. We saved ~45ms per query just by keeping connections warm. If you're using serverless, this is table stakes.
  • $ Feature-flag everything, even infra. We rolled out edge caching one route at a time via LaunchDarkly. Caught a serialization bug in /api/v2/allergies before it hit prod traffic.
  • $ The best optimization is the one you can ship this week. A 3-week infra project beat a 6-month rewrite. Not always the right call, but always worth asking.
