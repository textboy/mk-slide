03 — Data Flow

Event-Driven, End to End

① Ingest
Debezium captures row-level changes from PostgreSQL WAL. Zero app-code changes.
② Process
Flink SQL windows, aggregates, and joins across event streams with exactly-once guarantees.
③ Serve
ClickHouse materializes views for sub-100ms analytical queries. Redis for hot path.
Average end-to-end latency: 340ms (was 10+ minutes)
04 — Implementation

Flink SQL: Real-Time Aggregation

fraud_detection.sql Flink SQL
SELECT user_id, COUNT(*) AS tx_count, SUM(amount) AS total_amount, TUMBLE_START(event_time, INTERVAL '5' MINUTE) AS window_start FROM transactions WHERE event_time > CURRENT_TIMESTAMP - INTERVAL '1' HOUR GROUP BY user_id, TUMBLE(event_time, INTERVAL '5' MINUTE) HAVING COUNT(*) > 20 OR SUM(amount) > 10000

$ Tumbling windows flag suspicious activity within 5-minute intervals. Alerts fire in <1s.

07 — Lessons

What We Learned the Hard Way

lessons.log
1. Schema evolution breaks everything. Use Avro + Schema Registry from day one.
2. Exactly-once is a spectrum. Understand your actual consistency needs before over-engineering.
3. Monitoring is not optional. Consumer lag alerts saved us more than any code review.
4. Start with Flink SQL. Drop to Java only when SQL can't express your logic.
5. Backpressure is a feature. Don't fight it. Instrument it, tune it, respect it.
08 — Config

Production Kafka Tuning

kafka-config.yml YAML
# Broker — optimized for throughput + durability num.partitions: 12 replication.factor: 3 min.insync.replicas: 2 compression.type: lz4 # Producer — exactly-once semantics acks: all enable.idempotence: true max.in.flight.requests: 5 # Consumer — backpressure-aware max.poll.records: 500 fetch.max.wait.ms: 100 auto.offset.reset: earliest

$ These settings sustain 1.2M events/s across 3 AZs with p99 latency under 50ms.
